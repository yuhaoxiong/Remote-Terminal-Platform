> 状态：设计阶段

## 1. 项目概述

### 1.1 背景

在多地部署了基于 Debian 11 的 AI 边缘设备，通过 frp 隧道与云服务器建立连接，并反向代理到个人设备进行访问。当前缺少统一的管理平台来管理这些分散的边缘设备，设备注册、端口分配、状态监控、远程操作、批量更新均依赖手动操作，效率低且易出错。

### 1.2 目标

构建一个 **Web 端远程管理平台**，实现：

- 边缘设备自动注册与 frp 端口自动分配
- 设备在线状态实时监控与系统资源采集
- Web 端远程 SSH 和 VNC 操作
- 按项目号/程序类型批量远程更新
- 设备分组管理、文件传输、定时任务等辅助功能

### 1.3 技术栈

| 层级 | 技术选型 |
|------|----------|
| 后端框架 | Python 3.11+ / FastAPI |
| 前端框架 | Vue 3 + Element Plus + TypeScript |
| 数据库 | SQLite（单文件，通过 SQLAlchemy ORM） |
| Web SSH | xterm.js + WebSocket + paramiko |
| Web VNC | noVNC + WebSocket 代理 |
| 任务调度 | APScheduler |
| 文件传输 | paramiko SFTP |
| 认证 | JWT Token（单用户） |
| 部署 | 与 frps 同一台云服务器 |

### 1.4 边缘设备环境

| 项目 | 说明 |
|------|------|
| 操作系统 | Debian 11 |
| 隧道 | frpc 客户端（独立配置） |
| VNC | TigerVNC / TightVNC |
| SSH | OpenSSH Server |
| 管控方式 | 纯 SSH 远程执行（无边缘端 Agent） |

### 2.2 核心通信流程

#### 设备注册流程

```
管理员 ──► 平台Web ──► FastAPI ──► 分配端口 ──► 写入SQLite
                                       │
                                       ▼
                               生成frpc配置片段
                                       │
                                       ▼
                          SSH连接设备 ──► 写入frpc.toml ──► 重启frpc
```

#### 远程SSH流程

```
管理员浏览器(xterm.js) ◄──WebSocket──► FastAPI ◄──paramiko SSH──► frps(端口X) ◄──frp隧道──► 边缘设备SSH(22)
```

#### 远程VNC流程

```
管理员浏览器(noVNC) ◄──WebSocket──► FastAPI(websockify) ◄──TCP──► frps(端口Y) ◄──frp隧道──► 边缘设备VNC(5901)
```

### 2.3 端口分配策略

| 服务 | 端口范围 | 说明 |
|------|----------|------|
| SSH隧道 | 10000 - 10499 | 每台设备分配1个，映射到设备本地22端口 |
| VNC隧道 | 10500 - 10999 | 每台设备分配1个，映射到设备本地5901端口 |
| 预留扩展 | 11000 - 11999 | 未来可能的其他服务端口 |

最大支持 500 台设备（远超当前 50-200 台需求）。端口分配采用自增策略，已删除设备的端口可回收复用。

### 3.3 设备管理模块

**功能描述**：管理边缘设备的完整生命周期。

#### 3.3.1 设备注册

**流程**：

1. 管理员在平台填写设备信息（名称、IP/域名、SSH凭据、项目号、部署地点等）
2. 平台自动从端口池中分配未使用的 SSH 和 VNC 端口
3. 平台通过 SSH 连接到设备（首次需要设备可达，可通过临时端口或已有通道）
4. 平台生成 frpc 配置片段并写入设备
5. 平台重启设备上的 frpc 服务
6. 设备信息和端口映射关系写入数据库

**首次注册的连接问题**：

新设备首次注册时尚未建立frp隧道，平台无法通过frp连接。解决方案：

- **方案A**：设备预装一个最小frpc配置（仅注册用SSH端口），注册后平台替换为完整配置
- **方案B**：设备在局域网内可达时直接通过内网IP注册
- **方案C**：手动将frpc配置文件拷贝到设备后，再在平台录入信息

推荐方案A，设备出厂预装包含注册脚本，脚本调用平台API完成自动注册。

### 3.4 在线监控模块

**功能描述**：实时检测设备在线状态。

**实现方案**：

平台通过定时任务（APScheduler）周期性检测设备状态：

1. **TCP端口探测**：尝试连接设备的 frp SSH 端口（如 10001），连通则在线
2. **SSH连接检测**：尝试建立 SSH 连接并执行 `echo ok`，成功则完全可用
3. **检测频率**：默认每 60 秒一次，可配置
4. **状态判定**：
   - `online`：SSH连接正常
   - `offline`：TCP端口不通（连续3次失败后判定）
   - `degraded`：TCP通但SSH认证失败

**前端展示**：
- 设备列表中显示在线/离线状态标识（绿色/红色/灰色圆点）
- 支持按状态筛选
- 状态变更时实时更新（WebSocket推送或轮询）

### 3.5 远程SSH模块

**功能描述**：在 Web 浏览器中直接进行 SSH 终端操作。

**技术方案**：

```
浏览器 xterm.js ◄──WebSocket──► FastAPI WebSocket端点 ◄──paramiko──► 设备SSH
```

**实现细节**：

1. 前端使用 `xterm.js` + `xterm-addon-fit` 渲染终端
2. 建立 WebSocket 连接到后端 `/ws/ssh/{device_id}`
3. 后端使用 `paramiko` 通过 frp 隧道建立 SSH 连接
4. 双向转发：xterm.js 输入 → paramiko channel → 设备；设备输出 → paramiko channel → xterm.js
5. 支持终端大小自适应（resize事件同步）
6. 连接超时自动断开（可配置，默认30分钟无操作）

**接口**：

| 方法 | 路径 | 说明 |
|------|------|------|
| WebSocket | `/ws/ssh/{device_id}` | SSH终端WebSocket连接 |

### 3.6 远程VNC模块

**功能描述**：在 Web 浏览器中直接进行 VNC 远程桌面操作。

**技术方案**：

```
浏览器 noVNC ◄──WebSocket──► FastAPI(websockify代理) ◄──TCP──► frps VNC端口 ◄──frp──► 设备VNC(5901)
```

**实现细节**：

4. 支持 VNC 认证（密码转发）
5. 支持剪贴板同步
6. 支持分辨率自适应

**接口**：

| 方法 | 路径 | 说明 |
|------|------|------|
| WebSocket | `/ws/vnc/{device_id}` | VNC代理WebSocket连接 |

### 3.7 批量更新模块

**功能描述**：按项目号或程序类型对设备进行批量远程更新。

#### 3.7.1 更新任务类型

| 类型 | 说明 | 典型操作 |
|------|------|----------|
| 程序更新 | 更新AI推理程序或模型文件 | 上传文件 → 停止服务 → 替换文件 → 启动服务 |
| 系统更新 | 更新系统包和配置 | apt update → apt upgrade → 重启服务 |
| 脚本执行 | 执行自定义脚本 | 上传脚本 → chmod +x → 执行 → 返回结果 |
| 配置同步 | 同步配置文件 | 上传配置 → 备份旧配置 → 替换 → 重启服务 |

#### 3.7.2 更新流程

1. 创建更新任务：选择目标设备（按项目号/分组/标签/手动选择）
2. 配置更新内容：选择更新类型，上传文件或编写命令
3. 设置执行策略：
   - **并发数**：同时执行的设备数量（默认5）
   - **失败策略**：继续 / 暂停 / 回滚
   - **超时时间**：单设备执行超时（默认300秒）
4. 执行更新：逐批通过 SSH 执行更新操作
5. 实时进度：WebSocket 推送每台设备的执行状态和输出
6. 结果汇总：成功/失败/跳过设备统计

#### 3.7.3 更新任务状态

```
created → running → completed
                 → partial_failed（部分失败）
                 → failed（全部失败）
                 → cancelled（手动取消）
```

**接口**：

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/update-tasks` | 获取更新任务列表 |
| POST | `/api/update-tasks` | 创建更新任务 |
| GET | `/api/update-tasks/{id}` | 获取任务详情（含各设备执行状态） |
| POST | `/api/update-tasks/{id}/execute` | 执行更新任务 |
| POST | `/api/update-tasks/{id}/cancel` | 取消更新任务 |
| WebSocket | `/ws/update-tasks/{id}` | 实时更新进度推送 |

### 3.9 系统资源监控模块

**功能描述**：定期采集边缘设备的系统资源信息并展示。

**采集指标**：

| 指标 | 采集命令 | 频率 |
|------|----------|------|
| CPU使用率 | `top -bn1 \| grep Cpu` | 5分钟 |
| 内存使用 | `free -m` | 5分钟 |
| 磁盘使用 | `df -h` | 30分钟 |
| CPU温度 | `cat /sys/class/thermal/thermal_zone*/temp` | 5分钟 |
| 系统负载 | `cat /proc/loadavg` | 5分钟 |
| 运行时长 | `uptime -s` | 30分钟 |
| GPU信息 | 根据设备类型选择命令（nvidia-smi等） | 5分钟 |

**实现方案**：

1. APScheduler 定时任务触发采集
2. 通过 SSH 并发连接各设备执行采集命令
3. 解析输出结果存入数据库
4. 保留最近7天的详细数据，超过7天的按小时聚合保留30天
5. 前端使用 ECharts 展示趋势图

**接口**：

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/devices/{id}/metrics` | 获取设备资源指标 |
| GET | `/api/devices/{id}/metrics/latest` | 获取最新一次采集结果 |
| GET | `/api/devices/{id}/metrics/history` | 获取历史趋势数据 |

### 3.11 文件传输模块

## 9. 安全设计

### 9.1 认证与授权

- JWT Token 认证，有效期24小时，支持刷新
- 密码使用 bcrypt 哈希存储
- 所有 API 和 WebSocket 连接需验证 Token

### 9.2 数据安全

- SSH 密码和私钥使用 AES-256 加密存储在数据库中
- 加密密钥通过环境变量或配置文件注入
- SQLite 数据库文件权限设为 600

### 9.3 通信安全

- 平台使用 HTTPS（Nginx + Let's Encrypt）
- frp 隧道建议开启 TLS 加密
- WebSocket 通过 WSS 传输

### 9.4 操作安全

- 所有远程操作记录操作日志
- 批量更新操作需要二次确认
- 危险命令（rm -rf, shutdown 等）可选警告
