# 边缘设备功能生命周期平台执行计划

> 阶段：`approved_execution_plan`
> 状态：阶段 1、2 已完成；阶段 3 第一批（标准包与制品发布）已完成，下一批进入单设备部署计划
> 决策日期：2026-07-17
> 适用基线：`main@2e1dec2`

## 1. 目标

平台从“远程终端与通用 SSH 任务工具”演进为“以设备为中心的边缘功能生命周期平台”，统一管理：

- 业务项目、可复用功能、不可变版本和硬件变体；
- 新设备的 FRP、SSH、x0vncserver 管理基线；
- 标准功能包上传、发布、人工审批、设备下载、安装、检查和回滚；
- RK3568-4G 与 RK3588-8G 两类 Debian 11 设备；
- 小批量人工部署、小时级只读巡检及分级告警。

平台不接收识别结果、现场图片或其他业务事件。业务功能继续把数据发送到独立业务平台。

## 2. 冻结领域语言

| 术语 | 定义 |
|---|---|
| 项目 Project | 业务、现场或客户项目；一台设备至多属于一个项目 |
| 功能 Function | 可跨项目复用的独立能力，例如“桶外垃圾袋识别” |
| 版本 Release | 功能的一次不可变发布 |
| 硬件规格 Hardware Profile | 平台认可的设备规格枚举 |
| 变体 Variant | 同一版本面向某一硬件规格的制品与运行配置 |
| 项目功能 Project Function | 项目启用的功能及其期望版本、项目级默认配置 |
| 单设备覆盖 Device Override | 有原因、创建人和到期时间的临时版本覆盖 |
| 部署计划 Deployment Plan | 人工确认前冻结的设备、版本、变体、配置与哈希快照 |
| 部署执行 Deployment Execution | 确认计划后产生的幂等执行实例 |
| 初始化包 Bootstrap Bundle | 现场人员经局域网或 U 盘交付的一次性单设备在线安装包 |

设备角色字段仅预留，不参与第一阶段功能选择。项目下所有设备默认安装该项目启用的全部功能。

## 3. 核心决策

### 3.1 项目与设备

1. `Project` 升级为正式实体，设备通过可空外键关联项目。
2. 旧设备的自由文本项目号不迁移，升级时清空项目关系，由管理员重新分配。
3. 未分配项目的设备可完成初始化和硬件采集，但不能部署业务功能。
4. 已部署设备更换项目只生成迁移计划，不自动安装或卸载。
5. 从项目移除功能只标记待卸载；普通卸载保留配置和数据，破坏性清理单独确认。

### 3.2 设备初始化

1. 新建设备后自动生成初始化包草稿；字段校验通过后允许下载。
2. 每台设备独立生成，不共享令牌、VNC 密码或设备身份。
3. Bash 为入口，可执行 `apt-get update` 并安装白名单内缺失软件；不替换软件源、不执行系统全量升级。
4. 初始化安装 FRP、SSH、x0vncserver、轻量 `edge-deploy` CLI、硬件采集和芯片性能 governor oneshot 服务。
5. SSH 成功是注册硬门槛；VNC 可处于“待修复”。初始化脚本绝不自动重启，只返回 `ready`、`reboot_required` 或 `failed`。
6. 设备主动访问平台 HTTPS。过渡期使用自建 CA、带 IP SAN 的证书，并在包中携带 CA 公钥和指纹；禁止关闭 TLS 校验。
7. 一次性注册令牌成功后永久失效；失败步骤可幂等重试。关键连接信息变更后旧包失效，必须显式重新生成。
8. 人工序列号是可读身份，平台 UUID 是不可变内部身份；machine-id 和 MAC 仅作辅助指纹。

### 3.3 功能包与运行时

1. 第一阶段采用 Python 虚拟环境 + systemd，每个功能拥有独立目录、虚拟环境和服务；预留 Docker Compose 驱动。
2. 程序、配置、数据分离：
   - `/opt/edge-apps/<function>/<version>`
   - `/etc/edge-apps/<function>`
   - `/var/lib/edge-apps/<function>`
3. 升级保留设备端人工调试配置。默认配置只用于首次安装；配置结构变化必须由包提供迁移脚本。
4. 每个标准包必须提供 `manifest.yaml`、生命周期脚本和 `healthcheck.sh`。巡检采集由 `collect-schema.json` 与 `collect.sh` 声明。
5. 平台只接收预构建制品，不构建算法源码。制品发布后版本和 SHA-256 不可变。
6. 设备通过 HTTPS 主动下载，支持 Range、断点续传、重试和 SHA-256 校验。
7. 第一阶段采用 HTTPS + SHA-256；清单预留 `signature` 和 `key_id`，后续升级 Ed25519 签名。

### 3.4 部署与审批

1. 部署入口位于项目页面：解析项目功能、设备规格和临时覆盖，生成计划后必须由管理员确认。
2. 计划保存设备、版本、变体、配置、制品哈希与预检结果快照，有效期 24 小时；相关状态变化立即使其过期。
3. 确认后生成唯一 `execution_id`。设备持久化执行记录；重复 ID 只能返回或续跑已有执行，不得重复安装。
4. 生产设备部署失败自动回滚；测试设备保留失败现场，默认不回滚且不产生例行巡检告警。
5. 先在单台测试设备完成纵向验收，再开放最多 5 台并发。各设备隔离执行，单台失败不阻止其他设备。
6. 定时任务只用于小时级只读巡检；部署主要由人工发起。已批准计划可预约单次执行，不允许周期性自动部署或自动纠偏。

### 3.5 安全与审计

1. 包上传、发布、部署、版本覆盖和破坏性操作仅管理员可执行；操作员只读并可预览。
2. 密码与令牌在数据库中加密保存，主密钥独立提供；页面、日志和采集结果必须脱敏。
3. 设备端敏感配置归 `root:root`，权限 `0600`。
4. 所有审批、覆盖、迁移、部署、回滚和卸载均记录操作者、原因、目标、前后状态和时间。

## 4. 硬件规格

第一阶段只启用两个显式规格，不建设通用规则引擎：

| 规格代码 | SoC | 内存 | 系统 | RKNN |
|---|---|---:|---|---|
| `rk3568-4g-debian11` | RK3568 | 4 GB | Debian 11 | 当前统一版本 |
| `rk3588-8g-debian11` | RK3588 | 8 GB | Debian 11 | 当前统一版本 |

设备创建时保存期望规格，初始化时采集实际规格。两者不符时允许完成管理基线，但阻止业务部署，直至管理员确认。

## 5. 数据模型路线

阶段 1 引入：

- `projects`
- `hardware_profiles`
- `functions`
- `function_releases`
- `function_variants`
- `project_functions`
- `device_release_overrides`
- `deployment_plans`
- `deployment_plan_items`
- `deployment_executions`
- `deployment_execution_items`

设备表调整：

- `project_id`：可空正式外键，旧字符串值清空；
- `expected_profile_id`：管理员选择的期望规格；
- `actual_profile_id`：初始化后解析的实际规格；
- `device_uuid`：不可变且唯一；
- `device_role`：预留但不参与部署；
- `is_test_device`：控制验收、回滚和告警策略；
- 初始化状态、辅助指纹和最近采集事实按后续阶段增加。

## 6. 分阶段执行

### 阶段 1：领域基础

- 新增正式项目、功能、版本、变体、设备规格和部署计划模型。
- 把设备项目号改为可空外键，升级时清空旧关系。
- 提供管理员项目/功能/版本管理 API，设备创建支持未分配及正式项目选择。
- 冻结发布版本不可修改、项目只能绑定已发布版本、硬件变体唯一等领域约束。
- 保留通用 SSH 更新任务，但项目筛选改用正式项目 ID。

验证：迁移测试、领域服务测试、API 权限测试、设备 CRUD 回归、全量后端测试。

### 阶段 2：初始化闭环

- 单设备初始化包草稿、校验、下载、失效与重新生成。
- 一次性注册令牌、CA 指纹、FRP/SSH/VNC 配置和密码生成。
- Bash 安装器、`edge-deploy` CLI、硬件采集和注册 API。
- 期望/实际规格校验、SSH 硬门槛、VNC 可修复状态。

验证：脚本静态检查、包内容安全测试、令牌幂等测试、Debian 11 全新设备验收。

### 阶段 3：单设备部署闭环

- 本地制品仓库和 Nginx 下载端点。
- 清单校验、草稿上传、不可变发布和制品 SHA-256。
- 项目部署计划解析、24 小时失效、人工确认和唯一执行 ID。
- SSH 调用 `edge-deploy`，设备下载并执行 preflight/install/configure/start/healthcheck。
- 生产失败回滚、测试设备保留现场、全量审计。

验证：单台测试设备从上传到健康通过的端到端验收。

### 阶段 4：桶外垃圾袋识别重构

- 算法源码保留为独立 Git 仓库；平台只保存制品。
- 移除功能包内 FRP、VNC 和通用设备基线职责。
- 复用现有 release、activation、rollback、runtime health/acceptance 能力，用标准生命周期脚本做薄适配。
- 输出 RK3568 与 RK3588 两个变体：
  - RK3568：`best1.rknn`，`rubbish` 原始类别 0；
  - RK3588：`best.rknn`，`rubbish` 原始类别 1。
- 根据两台真机采集的输出数量、形状和值域决定各自后处理；不得继续依靠手改同一 `demo.py`。
- 生产运行采用无界面 systemd；PySide6 GUI 仅供 VNC 人工调试，并阻止与服务同时占用摄像头。
- 识别事件和图片仍由功能发送到另一业务平台，平台不保存业务数据。

验证：两类真机实时识别、固定样本证据、安装、重启、健康检查和回滚全部通过。

### 阶段 5：批量部署、巡检与告警

- 开放最多 5 台并发，设备间失败隔离。
- 每个功能按小时随机错峰执行 `healthcheck.sh` 和只读采集。
- 离线只通知；在线功能异常按连续失败次数分级；部署后失败立即处理；测试设备免例行告警。
- 单设备临时覆盖展示漂移、到期和审计。
- 交付通用功能包模板、重构指南和运维手册。

## 7. 标准功能包外层契约

```text
<function>-<version>-<variant>/
├── manifest.yaml
├── artifacts/
├── config/
│   ├── default.yaml
│   └── collect-schema.json
├── scripts/
│   ├── preflight.sh
│   ├── install.sh
│   ├── configure.sh
│   ├── start.sh
│   ├── healthcheck.sh
│   ├── collect.sh
│   ├── rollback.sh
│   └── uninstall.sh
└── systemd/
    └── service.template
```

生命周期脚本必须非交互、可重复运行、返回明确退出码，并输出约定 JSON。`healthcheck.sh` 至少返回功能状态、检查项、严重级别和时间。详细 JSON Schema 在阶段 3 固化。

## 8. 最终验收

- [ ] 现有设备、端口、凭据和任务数据保留，仅清空旧项目关系。
- [ ] Debian 11 新设备可用独立初始化包完成管理基线与平台注册。
- [ ] 标准包可上传、发布、预览计划、人工确认、部署、检查和回滚。
- [ ] RK3568-4G 与 RK3588-8G 真机均通过“桶外垃圾袋识别”实时测试。
- [ ] 平台不存储识别事件或图片。
- [ ] 最多 5 台并发部署、小时级巡检和分级告警生效。
- [ ] 通用模板、完整示例、部署/运维/回滚文档和自动化测试齐全。
- [ ] 所有生产部署及破坏性操作有人工确认和审计。

## 9. 实施纪律

1. 每个阶段先冻结失败测试和 API 契约，再做最小实现。
2. 每阶段独立提交、独立验证，后一阶段不混入前一阶段未完成改动。
3. 不删除或覆盖用户现场调试配置和业务数据。
4. 不在平台仓库提交模型测试图片、现场识别结果、密钥或一次性初始化包。
5. 两类真机证据不进入 Git，只在本地验收目录保存哈希与结论。
