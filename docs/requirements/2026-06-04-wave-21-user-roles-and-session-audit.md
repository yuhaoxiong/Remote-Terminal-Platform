# Wave 21 需求文档：多管理员账号、角色权限与会话审计基础

> 阶段：`requirement_doc`
> 状态：已确认，已按执行计划实现并进入验证收尾
> 基线：Wave 20 已完成告警外部通知、Webhook 通道、通知策略、投递记录、失败重试、诊断摘要和前端配置入口。平台已经具备远程连接、文件管理、真实 SSH 批量任务、定时调度、告警中心和外部通知能力，下一阶段应把“单管理员全权限测试平台”推进为“多管理员可审计运维平台”的基础形态。

## 1. 背景

截至 Wave 20，平台已经具备：

- 管理员登录、JWT access/refresh token、修改密码和前端自动 refresh。
- 设备 CRUD、分组、标签、frps Dashboard 导入、设备级 SSH 凭据、凭据加密和 known_hosts 策略。
- Web SSH、VNC、设备文件管理、设备状态、设备指标、监控总览和系统诊断。
- 批量更新任务：目标预览、命令模板、dry-run、真实 SSH 执行、失败设备重试、结果 CSV 导出和 WebSocket 快照。
- 定时任务：配置管理、自动调度、立即执行、执行记录和调度器状态。
- 告警中心：设备状态、指标阈值、指标冻结、定时任务失败和批量任务失败告警，支持确认、恢复、规则启停和阈值编辑。
- 告警外部通知：Webhook 通道、通知策略、投递记录、失败重试和诊断摘要。
- 操作日志、README、API 文档、部署文档、前端手工测试指南和 Postman Collection。

当前平台的主要权限边界仍然较粗：

- 系统只有默认管理员账号，实际多人测试时容易共享同一密码。
- API 只判断“是否登录”，不区分普通运维、远程操作、配置管理和系统管理权限。
- Web SSH、VNC、文件上传/下载、真实 SSH 批量任务、定时任务和告警通知配置都属于高风险操作，但当前都由同一管理员权限覆盖。
- 操作日志已有用户 ID，但缺少用户管理、角色上下文和会话审计入口，追责能力不足。
- 后续如果直接做完整 RBAC、项目权限、设备范围授权和审批流，范围会过大，不适合一次完成。

Wave 21 的目标是先建立轻量多用户和角色权限基础，让平台至少能够区分“超级管理员”和“运维人员”，并把登录会话、远程会话和高风险操作纳入可审计范围。

## 2. 目标

Wave 21 完成后，系统应达到以下状态：

1. 平台支持多个后台用户账号，不再只能依赖默认 `admin`。
2. 用户具备启用/停用状态，停用用户不能登录，已有 refresh token 应失效或无法继续刷新。
3. 平台至少支持两个角色：`admin` 和 `operator`。
4. 高风险功能具备最小权限边界，普通运维不能执行系统管理类操作。
5. 前端新增用户管理入口，超级管理员可创建、编辑、停用用户和重置密码。
6. 操作日志和诊断摘要能体现用户与权限相关信息，便于部署后检查默认账号和用户状态。
7. 登录、刷新、退出或会话失效等认证事件有审计记录。
8. Web SSH/VNC、文件操作、真实 SSH 批量任务、定时任务执行、告警规则和通知配置等关键操作继续写入操作日志，并能关联用户。
9. API、迁移、前端、测试、文档和 Postman 验收路径完整闭环。
10. 所有新增前端文案继续使用中文。

## 3. 非目标

本轮不实现以下内容：

- 不实现完整企业级 RBAC，不支持任意权限点自定义组合。
- 不实现项目级、分组级、设备级数据范围授权。
- 不实现审批流、双人复核、工单流或命令执行前审批。
- 不实现 SSO、LDAP、OAuth、OIDC、MFA 或短信验证码。
- 不实现用户头像、个人资料、复杂密码策略配置页。
- 不实现多租户隔离。
- 不实现远程 SSH/VNC 录屏、完整终端回放或文件内容审计。
- 不改变当前 SQLite 单实例部署基线。

## 4. 角色与权限边界

### 4.1 角色定义

本轮建议内置固定角色：

| 角色 | 说明 |
| --- | --- |
| `admin` | 超级管理员，可管理用户、系统配置、设备、远程连接、任务、告警和通知 |
| `operator` | 运维人员，可查看设备、执行授权的运维操作，但不能管理用户和系统级配置 |

角色字段建议保存在 `users.role`。旧数据库升级时：

- 现有默认管理员用户自动设为 `admin`。
- 如果已有其他用户，默认设为 `operator` 或按迁移脚本明确处理。

### 4.2 权限矩阵

建议本轮先使用固定权限矩阵，不做数据库权限点表。

| 功能 | `admin` | `operator` |
| --- | --- | --- |
| 查看仪表盘、设备、指标、告警、日志、诊断 | 允许 | 允许 |
| 创建设备、编辑设备、删除设备 | 允许 | 待确认 |
| 打开 Web SSH/VNC | 允许 | 允许 |
| 文件列表、上传、下载、删除 | 允许 | 允许 |
| 创建 dry-run 批量任务 | 允许 | 允许 |
| 创建真实 SSH 批量任务 | 允许 | 待确认 |
| 取消批量任务、失败设备重试 | 允许 | 待确认 |
| 创建、编辑、删除定时任务 | 允许 | 待确认 |
| 手动执行定时任务 | 允许 | 待确认 |
| 编辑告警规则 | 允许 | 不允许 |
| 确认/恢复告警 | 允许 | 允许 |
| 管理告警通知通道和策略 | 允许 | 不允许 |
| 导出日志、导出任务结果 | 允许 | 允许 |
| 用户管理、重置密码、停用用户 | 允许 | 不允许 |
| 修改自己的密码 | 允许 | 允许 |

默认建议：

- `operator` 可以执行日常设备操作和告警处理。
- 涉及平台配置、通知通道、用户管理和删除类管理操作默认只允许 `admin`。
- 真实 SSH 批量任务、定时任务管理是否开放给 `operator` 作为待确认问题。

## 5. 用户管理需求

### 5.1 用户模型扩展

建议扩展 `users` 表：

```text
id
username
password_hash
role
is_active
last_login_at
last_login_ip
password_changed_at
created_at
updated_at
```

要求：

- `username` 继续唯一。
- `role` 必须使用枚举或等价校验。
- `is_active=false` 时不得登录。
- API 响应不得返回 `password_hash`。
- 密码仍使用当前哈希方案，不明文保存。
- 旧库迁移必须保持默认管理员可继续登录。

### 5.2 用户管理 API

建议新增：

```http
GET /api/users
POST /api/users
GET /api/users/{user_id}
PUT /api/users/{user_id}
POST /api/users/{user_id}/reset-password
POST /api/users/{user_id}/toggle
DELETE /api/users/{user_id}
```

语义要求：

- 以上接口除当前用户修改自己密码外，均要求 `admin` 角色。
- 创建用户时必须提供用户名、初始密码和角色。
- 编辑用户支持修改角色和启用状态。
- 重置密码只返回操作成功，不返回密码哈希。
- 停用用户后，该用户不能继续刷新 token。
- 删除用户建议采用软删除或禁用优先，避免破坏历史操作日志关联。
- 禁止停用或删除最后一个启用状态的 `admin` 用户。
- 禁止当前登录用户直接删除自己；可允许修改自己的密码。

### 5.3 认证响应扩展

当前 `GET /api/auth/me` 建议扩展：

```json
{
  "id": 1,
  "username": "admin",
  "role": "admin"
}
```

登录响应可保持 Token 结构不变，但前端应在登录后通过 `me` 或 Token 解析获取角色信息。

## 6. 权限校验需求

### 6.1 后端权限校验

后端需要提供统一角色校验 helper，避免每个 router 自己写字符串判断。

建议：

- `require_admin_user`
- `require_operator_user`
- `require_roles(["admin", "operator"])`

要求：

- 未登录继续返回 `401` 或当前认证机制对应错误。
- 登录但角色不足返回 `403`。
- 角色非法或用户停用时拒绝访问。
- 权限失败应写入操作日志或安全审计日志，至少记录动作、用户、目标和失败原因。

### 6.2 关键接口收敛

本轮至少应对以下接口加权限边界：

- 用户管理接口：仅 `admin`。
- 告警规则编辑：仅 `admin`。
- 告警通知通道/策略管理：仅 `admin`。
- 设备删除：仅 `admin`。
- 分组删除：仅 `admin`。
- 定时任务删除：仅 `admin`。
- 批量任务真实 SSH 执行：按待确认决策限制。
- 系统诊断：`admin` 和 `operator` 均可查看，但敏感 warning 仍只返回非敏感摘要。

## 7. 会话与审计需求

### 7.1 认证事件审计

以下事件应写入操作日志：

- 登录成功。
- 登录失败。
- refresh token 成功刷新。
- refresh token 失败。
- 修改密码成功或失败。
- 用户被停用后尝试登录或刷新。

注意：

- 日志不得记录密码、Token、refresh token、密码哈希或请求头 Authorization。
- 登录失败可记录用户名和来源 IP，但不得记录密码。

### 7.2 会话摘要

本轮可不做完整会话表，但应至少在用户表或诊断中体现：

- `last_login_at`
- `last_login_ip`
- 用户数量。
- 启用用户数量。
- 启用 admin 数量。
- 停用用户数量。

如实现会话表，可作为增强项：

```text
auth_sessions
id
user_id
created_at
last_seen_at
expires_at
revoked_at
client_ip
user_agent
```

本轮不强制做 refresh token 黑名单；但停用用户后，刷新接口必须检查用户 `is_active`。

### 7.3 高风险操作审计

以下操作必须继续或新增操作日志：

- Web SSH/VNC 会话创建、连接失败和断开摘要。
- 文件上传、下载和删除。
- 真实 SSH 批量任务创建、执行、取消、导出。
- 定时任务创建、编辑、删除、启停、手动执行。
- 告警规则修改。
- 告警通知通道和策略创建、编辑、删除、测试、重试。
- 用户创建、编辑、停用、重置密码。

日志要求：

- 记录 `user_id`。
- 记录动作名、目标类型、目标 ID、状态和中文摘要。
- 不记录密码、Token、Webhook Header 值、完整 SSH 命令输出或文件内容。

## 8. 前端需求

### 8.1 用户管理页面

前端建议新增“用户管理”入口，仅 `admin` 可见。

页面能力：

- 用户列表：用户名、角色、启用状态、最近登录时间、最近登录 IP、创建时间。
- 新增用户：用户名、初始密码、角色。
- 编辑用户：角色、启用状态。
- 重置密码：输入新密码并提交。
- 停用/启用用户。

交互要求：

- 所有文案中文。
- 不展示密码哈希。
- 不在页面保存或回显旧密码。
- 禁止当前用户停用最后一个 `admin` 时显示中文错误。
- 对权限不足返回 `403` 时显示“当前账号无权限执行该操作”，不退出登录。

### 8.2 权限感知 UI

前端应根据当前用户角色隐藏或禁用部分入口：

- `operator` 不显示“用户管理”。
- `operator` 不显示或禁用告警通知配置编辑按钮。
- `operator` 不显示或禁用告警规则保存按钮。
- 对普通查询入口不做过度隐藏，避免影响运维观察。

说明：

- 前端隐藏只做体验优化，真正权限必须以后端校验为准。
- 如果后端返回 `403`，前端应显示当前区域错误，不应清理 Token 或跳回登录页。

## 9. 诊断与部署需求

`GET /api/diagnostics/config` 建议新增 `users` 摘要：

```json
{
  "users": {
    "total_count": 2,
    "active_count": 2,
    "admin_count": 1,
    "operator_count": 1,
    "disabled_count": 0,
    "warnings": []
  }
}
```

诊断 warning 建议：

- 仍只有默认 `admin` 用户。
- 启用 admin 数量为 0。
- 存在停用用户尝试刷新 token。
- 默认管理员密码仍在使用。

部署文档需要补充：

- 首次部署后创建第二个 `admin` 或至少创建独立个人账号。
- 禁止多人共享默认 `admin`。
- 停用离职或不再参与测试的账号。
- 导出操作日志时可按用户筛选。

## 12. 验收标准

Wave 21 通过验收需满足：

1. 旧数据库升级后默认管理员仍可登录。
2. admin 可创建、编辑、停用用户和重置密码。
3. operator 无法访问用户管理、告警规则编辑和通知配置管理接口。
4. operator 权限不足时返回 `403`，前端显示中文权限提示且不退出登录。
5. 停用用户不能登录，也不能通过 refresh token 继续获得新 token。
6. 认证事件、用户管理事件和高风险操作均有操作日志。
7. 诊断接口返回用户摘要且不泄露密码、Token 或哈希。
8. README、API 文档、部署文档、前端手工测试指南和 Postman 已更新。
9. 后端、前端测试、类型检查和构建通过。
10. 所有新增前端文案为中文。

## 13. 待确认问题

1. `operator` 是否允许创建设备、编辑设备和删除设备？建议：允许创建/编辑，删除仅 admin。
2. `operator` 是否允许创建和执行真实 SSH 批量任务？建议：本轮先允许 dry-run，真实 SSH 仅 admin。
3. `operator` 是否允许管理定时任务？建议：查看和手动 dry-run 允许，创建/编辑/删除仅 admin。
4. 删除用户采用软删除、停用优先，还是允许硬删除？建议：停用优先，不硬删除，保留操作日志关联。
5. 是否需要本轮新增退出登录接口和 refresh token 撤销表？建议：本轮不做完整 token 黑名单，只确保停用用户 refresh 失败。
6. 用户管理入口放在独立一级导航，还是放在系统诊断/系统设置区域？建议：新增“用户管理”一级入口，仅 admin 可见。
7. 测试完成后是否继续统一提交并推送到 `origin/main`？建议：是。

## 14. 已确认决策

1. `operator` 允许创建设备和编辑设备，删除设备仅 `admin`。
2. `operator` 本轮只允许创建/执行 dry-run 批量任务，真实 SSH 批量任务仅 `admin`。
3. `operator` 可查看定时任务和手动 dry-run，定时任务创建、编辑、删除仅 `admin`。
4. 用户删除采用停用优先，不做硬删除，保留操作日志关联。
5. 本轮不新增完整 refresh token 黑名单，只确保停用用户登录和 refresh 失败。
6. 用户管理作为一级导航入口，仅 `admin` 可见。
7. 测试完成后统一提交并推送到 `origin/main`。
