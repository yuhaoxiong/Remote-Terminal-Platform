# Wave 19 需求文档：告警中心与设备健康事件闭环

> 阶段：`requirement_doc`  
> 状态：已确认，执行计划已生成，等待计划批准  
> 基线：Wave 18 已完成定时任务真实调度、执行记录、调度器诊断和前端可观测入口。平台已经具备设备指标、批量任务结果、定时任务执行事实和操作日志，下一阶段应把“异常可见”推进为“异常可追踪、可确认、可恢复”的运维闭环。

## 1. 背景

截至 Wave 18，平台已经具备：

- 管理员登录、JWT access/refresh token、修改密码、前端自动 refresh。
- 设备 CRUD、分组、标签、frps Dashboard 导入、设备级 SSH 凭据、凭据加密、known_hosts 策略。
- Web SSH、VNC、设备文件管理、设备状态、设备指标、监控总览和系统诊断。
- 批量更新任务：目标预览、命令模板、dry-run、真实 SSH 执行、失败设备重试、结果 CSV 导出和 WebSocket 快照。
- 定时任务：配置管理、自动调度、立即执行、执行记录和调度器状态。
- 操作日志、README、API 文档、部署文档和 Postman Collection。

当前异常信息已经存在，但仍分散在多个入口：

- 仪表盘能展示异常设备摘要，但更偏即时概览，缺少事件留痕和处理状态。
- 设备指标能反映 CPU、内存、磁盘和采集时间，但缺少规则化阈值判断与告警聚合。
- 定时任务执行记录能看到失败，但管理员需要进入具体任务后才能发现。
- 批量任务结果能看到失败设备，但任务失败并不会进入统一运维事件入口。
- 操作日志记录“发生了什么”，但不适合作为“当前有哪些异常还没处理”的工作台。

Wave 19 的目标是新增轻量告警中心，把设备健康异常、任务失败和定时调度异常收敛到统一事件入口，并提供确认、恢复和筛选能力，为后续通知、权限和报表能力建立数据基础。

## 2. 目标

Wave 19 完成后，系统应达到以下状态：

1. 平台存在统一“告警中心”入口，管理员可查看当前活跃告警和历史告警。
2. 设备状态异常、设备指标超阈值、指标过期、定时任务失败和批量任务失败可形成告警事件。
3. 同一异常在持续存在时会聚合到同一活跃告警，不会在每次扫描时无限刷屏。
4. 告警具备未确认、已确认、已恢复等处理状态，并记录关键时间。
5. 告警支持按级别、来源、设备、任务、状态和时间筛选。
6. 仪表盘和系统诊断能展示告警摘要，帮助部署后快速判断运维风险。
7. 新增 API、数据模型、迁移、前端页面、测试、文档和 Postman 验收路径完整闭环。
8. 所有新增前端文案继续使用中文。

## 3. 非目标

本轮不实现以下内容：

- 不实现短信、邮件、Webhook、企业微信、钉钉等外部通知通道。
- 不实现多用户 RBAC、值班排班、升级策略、审批流或 SLA 计时。
- 不实现复杂规则编排、表达式 DSL、任意脚本告警规则或用户自定义告警插件。
- 不实现完整报表中心、趋势分析平台或告警统计大屏。
- 不实现跨节点告警去重和分布式事件总线；仍以当前 SQLite 单实例部署为基线。
- 不把操作日志替换为告警系统；两者继续承担不同职责。
- 不要求 Wave 19 自动修复设备异常；本轮只做发现、归并、确认和恢复闭环。

## 4. 功能需求

### 4.1 告警范围

本轮告警中心至少覆盖以下来源：

| 来源 | 告警示例 | 触发依据 |
| --- | --- | --- |
| 设备状态 | 设备离线、设备未知 | 设备当前状态 |
| 设备指标 | CPU 过高、内存过高、磁盘过高 | 最新指标与阈值 |
| 指标采集 | 指标过期、长时间无指标 | 最新指标时间 |
| 定时任务 | 自动执行失败 | `scheduled_task_runs` 失败记录 |
| 批量任务 | 批量任务部分失败或失败设备存在 | 更新任务状态与设备结果 |

要求：

- 每类告警必须有稳定 `source_type` 和可追踪 `source_id`。
- 设备类告警必须关联设备 ID。
- 任务类告警应关联对应任务 ID；若任务与设备结果相关，可附带失败设备摘要。
- 本轮默认先支持平台内建告警类型，不要求管理员自由创建任意新告警类型。

### 4.2 告警级别与状态

告警级别建议支持：

- `warning`
- `critical`

告警处理状态建议支持：

- `open`：异常仍存在，尚未确认。
- `acknowledged`：管理员已确认异常，异常仍可能存在。
- `resolved`：异常已经恢复或被管理员关闭。

要求：

- 活跃告警指 `open` 或 `acknowledged`。
- 恢复后的告警默认保留历史，不直接删除。
- 确认告警需要记录确认时间和确认说明。
- 恢复告警需要记录恢复时间；若由系统自动恢复，应能区分系统恢复。
- 后端状态值必须使用枚举或等价约束，非法值返回明确错误。

### 4.3 告警去重与聚合

告警系统不能把每次指标采集或每次后台扫描都变成一条新活跃告警。

建议为每个活跃告警生成稳定 `dedupe_key`，至少包含：

- 告警类型。
- 来源类型。
- 来源 ID。
- 必要维度，例如设备 ID、指标名或任务 ID。

聚合语义：

- 同一 `dedupe_key` 的异常再次触发时，更新当前活跃告警的最近触发时间、触发次数和最近摘要。
- 已恢复告警再次出现时，可新建新的告警周期，保留历史恢复记录。
- 告警内容不得因去重而丢失最近失败原因、指标值或任务摘要。

### 4.4 内建规则

Wave 19 先实现少量高价值内建规则。

建议默认规则：

1. 设备状态异常：
   - `offline` -> `critical`
   - `unknown` 或 `degraded` -> `warning`
2. 设备最新指标超阈值：
   - CPU 高于 `85%`。
   - 内存高于 `85%`。
   - 磁盘高于 `90%`。
3. 指标采集过期：
   - 最近指标超过 `10` 分钟。
   - 从未上报指标的设备只提示“暂无指标”，不直接生成告警。
4. 定时任务失败：
   - 自动调度执行记录为 `failed`。
5. 批量任务失败：
   - 更新任务状态为 `partial_failed` 或存在失败设备结果。

规则配置边界：

- 阈值应集中配置，不在前后端散落硬编码。
- 本轮开放平台级规则启停与阈值编辑能力，但不做复杂规则表达式编辑器。
- 规则默认值必须可用；即使尚未编辑配置，默认指标阈值仍按 CPU `85%`、内存 `85%`、磁盘 `90%`、指标过期 `10` 分钟生效。

### 4.5 触发与恢复时机

告警需要在“异常事实产生”后尽快进入事件入口，同时避免引入过重后台系统。

触发建议：

- 设备指标写入后，对该设备执行指标阈值判断。
- 指标过期、设备状态异常可由后台周期扫描判断。
- 定时任务失败在执行记录落库后触发告警。
- 批量任务执行收口后，对失败任务或失败设备摘要触发告警。

恢复建议：

- 新指标恢复正常后，指标类活跃告警可自动恢复。
- 设备状态回到在线后，设备状态类活跃告警可自动恢复。
- 定时任务后续成功执行后，自动恢复同一任务上一轮仍活跃的失败告警。
- 批量任务失败默认不因后续其他任务成功自动恢复，应由管理员确认或关闭，避免掩盖失败历史。

### 4.6 数据模型建议

建议新增告警实体和必要配置实体。

#### 4.6.1 alerts

建议字段：

```text

dedupe_key
alert_type
severity
status
source_type
source_id
device_id
title
summary
detail
first_triggered_at
last_triggered_at
trigger_count
acknowledged_at
acknowledged_note

```

要求：

- `detail` 只保存可审计摘要，不保存明文密码、Token、私钥、passphrase。
- 若与批量任务或定时任务关联，优先保存 ID 与摘要，不复制大量输出。
- 需要有适合 SQLite 的索引：状态、严重级别、来源、设备、最近触发时间、去重键。

#### 4.6.2 alert_rules

本轮开放规则配置，建议新增：

```text

rule_type
enabled
severity
threshold_value
window_minutes
created_at
updated_at
```

要求：

- 默认规则需在旧库升级后自动可用。
- 支持规则启停。
- 支持指标阈值与过期窗口编辑。
- 不支持用户编写任意表达式或脚本规则。

### 4.7 API 需求

建议新增接口：

```http
GET /api/alerts
GET /api/alerts/summary
POST /api/alerts/{id}/acknowledge
POST /api/alerts/{id}/resolve
GET /api/alert-rules
PUT /api/alert-rules/{id}
```

其中：

- `GET /api/alerts`
  - 支持分页。
  - 支持按 `status`、`severity`、`source_type`、`device_id`、`alert_type`、时间范围筛选。
- `GET /api/alerts/summary`
  - 返回活跃告警数、严重告警数、最近告警时间和按来源统计。
- `POST /api/alerts/{id}/acknowledge`
  - 接收可选确认说明。
- `POST /api/alerts/{id}/resolve`
  - 用于管理员手动关闭仍需归档的告警。
- `GET/PUT /api/alert-rules`
  - 返回和更新平台内建规则的启停状态、级别、阈值与窗口配置。

要求：

- 所有接口需要登录鉴权。
- 告警响应不得返回敏感凭据。
- 手动确认、手动恢复应写操作日志。
- API 应保持中文错误提示与现有前端风格一致。

### 4.8 前端告警中心

前端新增一级“告警中心”入口，面向管理员高频排查使用。

页面至少包含：

- 告警摘要：
  - 活跃告警数。
  - 严重告警数。
  - 未确认告警数。
  - 最近告警时间。
- 告警表格：
  - 严重级别。
  - 标题。
  - 来源。
  - 关联设备/任务。
  - 首次触发时间。
  - 最近触发时间。
  - 触发次数。
  - 状态。
  - 操作。
- 筛选区：
  - 状态。
  - 级别。
  - 来源。
  - 设备。
  - 关键词或时间范围。
- 处理动作：
  - 确认告警。
  - 查看关联设备或任务。
  - 手动恢复告警。
- 规则配置区：
  - 查看平台内建规则。
  - 启用/停用规则。
  - 编辑 CPU、内存、磁盘阈值和指标过期窗口。

要求：

- 告警行应突出 `critical` 与未确认状态，但不要把页面做成营销式大卡片堆叠。
- 页面失败时只显示局部错误，不应导致登录态闪退。
- 所有文案使用中文。

### 4.9 仪表盘与诊断摘要

仪表盘增强：

- 展示活跃告警摘要。
- 异常设备摘要应能跳转到告警中心或筛选结果。
- 不在仪表盘复制完整告警列表。

系统诊断增强：

- 返回告警模块状态摘要：
  - 是否启用。
  - 活跃告警数。
  - 严重告警数。
  - 最近告警评估时间。
  - 最近评估错误。
- 若告警评估异常，诊断页显示 warning。

## 5. 安全要求

- 告警详情不得保存或返回明文密码、Token、私钥、passphrase 或凭据加密密钥。
- 告警摘要不得直接复制完整 SSH 输出；如需要排查，应跳转到任务详情查看已有摘要。
- 告警确认和恢复是有审计意义的动作，必须写操作日志。
- 对告警详情字段要做长度限制，避免异常堆栈或远程输出把数据库撑大。
- 前端告警中心不得把敏感凭据暴露到 URL、表格、下载内容或浏览器日志。

## 6. 兼容性要求

- Wave 19 不破坏现有设备、监控、批量任务、定时任务 API。
- 仪表盘现有异常设备摘要继续可用，但可改为复用告警摘要或跳转告警中心。
- 诊断接口只新增向后兼容字段。
- 默认无告警历史的旧数据库升级后，平台应能继续正常启动。
- Alembic 迁移必须覆盖新告警表和必要索引。

## 7. 验收标准

Wave 19 只有在以下条件全部满足后才算完成：

- 指标异常、设备状态异常、定时任务失败和批量任务失败均能进入统一告警入口。
- 相同活跃异常不会在重复扫描时刷出大量重复告警。
- 告警中心可筛选、确认、恢复并查看关联来源。
- 仪表盘展示告警摘要并提供进入告警中心的路径。
- 系统诊断能展示告警模块摘要或评估状态。
- 告警相关迁移可执行。
- 后端全量 pytest 通过。
- 前端 Vitest 通过。
- 前端 build 通过。
- README、API、部署文档和 Postman 同步更新。

## 8. 风险与约束

- 告警系统一旦覆盖来源过多，容易在首版同时引入规则管理、通知、权限和报表，导致范围失控；Wave 19 必须先守住平台内告警闭环。
- 设备状态本身可能依赖手动更新或有限观测，告警准确性取决于现有状态和指标数据质量。
- 指标阈值过低会产生噪音，阈值过高会漏报；默认值需要在执行计划前冻结。
- 告警自动恢复语义如果定义不清，会导致管理员误以为历史失败已经被修复。
- 批量任务与定时任务都可能产生失败事件，执行计划需要避免在同一失败链路上重复创建多条等价告警。

## 9. 待确认问题

1. Wave 19 是否按“平台内告警中心”推进，暂不接入 Webhook、邮件或其他外部通知？
2. 默认指标阈值是否冻结为 CPU `85%`、内存 `85%`、磁盘 `90%`、指标过期 `10` 分钟？
3. 从未上报指标的设备是否直接产生告警，还是只在仪表盘/设备页提示“暂无指标”？
4. 定时任务后续成功执行后，是否自动恢复上一轮失败告警？
5. 本轮是否开放规则启停和阈值编辑 API/前端，还是先以内建规则和配置项完成闭环？
6. Wave 19 实现完成后是否继续统一提交并推送到 `origin/main`？

## 10. 已确认决策

根据用户确认，本轮冻结以下决策：

1. Wave 19 先按平台内告警中心推进，不接入 Webhook、邮件或其他外部通知。
2. 默认指标阈值为 CPU `85%`、内存 `85%`、磁盘 `90%`，指标过期窗口为 `10` 分钟。
3. 从未上报指标的设备只提示“暂无指标”，本轮不直接生成告警。
4. 同一定时任务后续成功执行后，自动恢复其上一轮仍活跃的失败告警。
5. 本轮开放规则启停和阈值编辑 API/前端，但不扩展到复杂规则表达式或脚本规则。
6. 需求确认后生成执行计划，计划批准后实现；测试完成后统一提交并推送到 `origin/main`。
