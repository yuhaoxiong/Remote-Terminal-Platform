## 1. 背景

平台截至 Wave 12 已完成登录鉴权、设备 CRUD、frps 导入、设备级 SSH 凭据、凭据加密、远程 SSH/VNC 基础代理、真实批量 SSH 任务、分组管理、日志筛选导出、系统诊断页、Postman 与部署文档。

当前监控能力的状态：

- 后端已有设备指标接口：
  - `POST /api/devices/{device_id}/metrics`
  - `GET /api/devices/{device_id}/metrics`
  - `GET /api/monitoring/overview`
- 指标模型已包含 `cpu_percent`、`memory_percent`、`disk_percent`、`recorded_at`。
- 前端仪表盘目前只使用 `GET /api/monitoring/overview` 的总览数据。
- 前端设备资源快照中的 CPU / 内存仍由本地 `mapDevice()` 固定为 `0`，没有读取真实指标。
- 诊断页已经展示安全摘要，但还没有展示监控数据是否可用。

Wave 13 的目标是先把“已有监控数据可见、无数据不误导、异常设备可识别”做完整，不引入告警系统、报表中心或复杂资产统计。

## 2. 目标

管理员登录平台后，应能在仪表盘上看到可信的设备运行态势：

1. 设备总数、在线数、离线/未知数继续来自后端总览。
2. 每台设备的最新 CPU、内存、磁盘指标来自真实指标接口。
3. 没有指标数据的设备显示“暂无指标”，不能继续显示 `0%` 造成误判。
4. 仪表盘展示异常设备摘要，便于快速定位离线、未知或资源过高设备。
5. 系统诊断页展示监控数据可用性，方便部署后判断设备是否已经上报指标。

## 3. 非目标

本轮不实现以下内容：

- 不做告警规则、告警通知、Webhook、邮件或机器人通知。
- 不做完整报表中心、月报、在线率统计报表。
- 不做后台定时采集器或 agent 安装器。
- 不做 GPU 指标采集。
- 不做多用户权限、角色可见范围或项目级权限。
- 不引入 Pinia / Vue Router 全量重构。
- 不强制接入真实生产设备；允许通过现有指标上报接口和测试数据验收。

## 4. 功能需求

### 4.1 前端 API 封装

需要在 `frontend/src/api/platform.ts` 中补齐指标接口类型和方法。

新增类型：

- `DeviceMetricRead`
  - `id`
  - `device_id`
  - `status`
  - `cpu_percent`
  - `memory_percent`
  - `disk_percent`
  - `recorded_at`
- `DeviceMetricListResponse`
  - `total`
  - `items`

新增方法：

- `listDeviceMetrics(deviceId: number, limit?: number): Promise<DeviceMetricListResponse>`

约束：

- 默认只取最近若干条，建议 `limit=20`。
- 前端仪表盘使用最新一条作为资源快照。
- 指标字段为 `null` 时必须按“暂无数据”处理。

### 4.2 仪表盘资源快照

当前仪表盘的“资源快照”需要接入真实指标。

展示要求：

- 每台设备显示：
  - 设备名称
  - 项目号
  - 当前状态
  - CPU
  - 内存
  - 磁盘
  - 最近指标时间
- 有指标时显示百分比和进度条。
- 无指标时显示“暂无指标”，不显示 `0%`。
- 指标时间为空时显示“未上报”。
- 最新指标超过 10 分钟未更新时显示“指标过期”提示。

资源阈值建议：

- CPU >= 90%：高负载
- 内存 >= 85%：高内存
- 磁盘 >= 90%：磁盘紧张

### 4.3 异常设备摘要

仪表盘新增“异常设备”区域。

纳入异常摘要的设备：

- `status` 为 `offline`
- `status` 为 `unknown`
- 最新 CPU >= 90%
- 最新内存 >= 85%
- 最新磁盘 >= 90%
- 最新指标超过 10 分钟未更新

展示要求：

- 最多显示前 8 条。
- 每条包含设备名称、项目号、异常类型和简短说明。
- 点击或按钮可跳转到设备管理页，并保留搜索/筛选入口。
- 没有异常时显示“暂无异常设备”。

### 4.4 基础可视化

仪表盘增加轻量图表，不做报表系统。

至少包含：

- 设备状态分布：
  - 在线
  - 离线
  - 未知
  - 异常
- 资源风险分布：
  - 正常
  - CPU 高
  - 内存高
  - 磁盘高
  - 无指标

实现约束：

- 本轮允许引入 ECharts。
- 如果引入 ECharts 导致 bundle 继续膨胀，需要在构建结果中说明。
- 图表区域在无数据时必须显示空状态。
- 图表不能替代文字摘要；关键数字必须仍以文本展示。

### 4.5 监控数据加载策略

登录后 `loadPlatformData()` 需要额外加载设备最新指标。

建议策略：

- 先加载设备列表和分组列表。
- 对设备列表中每台设备并发请求 `GET /api/devices/{id}/metrics?limit=1`。
- 本轮冻结为沿用每台设备 `limit=1` 查询最新指标，不新增批量最新指标接口。
- 单台设备指标加载失败时，不影响整页显示。
- 指标接口整体失败时，显示“指标加载失败”，但不踢回登录页，除非返回 401/403。
- 避免因为指标接口失败导致 Wave 8 曾出现过的“登录后闪退回登录页”问题复现。

### 4.6 诊断页监控可用性

系统诊断页需要补充监控数据可用性。

可选实现方式：

- 优先前端从已加载设备指标推导：
  - 有指标设备数
  - 无指标设备数
  - 最近指标时间
- 如果后端新增诊断字段，则必须保证不返回敏感信息。

本轮建议不新增复杂后端诊断结构，先在前端基于现有指标接口展示：

- “有指标设备：X”
- “无指标设备：Y”
- “最近指标时间：YYYY-MM-DD HH:mm”

### 4.7 中文显示

所有新增前端文案必须使用中文。

包括：

- 指标加载状态
- 空状态
- 异常说明
- 诊断摘要
- 测试中的可见文本断言

## 5. 后端需求

后端已有指标接口，本轮以复核和小幅增强为主。

必须保持：

- `POST /api/devices/{device_id}/metrics` 可写入 CPU、内存、磁盘和状态。
- `GET /api/devices/{device_id}/metrics` 可按时间倒序返回指标。
- `GET /api/monitoring/overview` 可返回设备状态总览。

需要复核：

- `GET /api/devices/{device_id}/metrics?limit=1` 能稳定返回最新一条。
- 无指标时返回 `total=0`、`items=[]`。
- 设备不存在时返回合理错误。
- 未登录访问返回 401 或 403。

如果当前后端已经满足以上行为，本轮不强制新增后端接口。

- 单设备指标失败：该设备显示“指标加载失败”。
