## 背景

当前平台已经完成设备管理、分组与日志、监控概览、批量更新任务模型、定时任务、文件管理、真实 SSH/VNC WebSocket 基础能力、SFTP 文件后端、frps Dashboard 自动发现与导入，以及前端中文化。

本轮测试已经覆盖了 frps 导入功能，但在服务器环境继续测试后端 API 时出现过 `502`，并暴露出几个需要固化为产品能力的问题：

- 服务器部署链路依赖手工排查，后端、前端代理、Nginx、端口监听、环境变量之间缺少一键诊断。
- Postman/OpenAPI API 验收需要更稳定的导入文件、登录后自动保存 Token、受保护接口自动携带 Bearer Token。
- frps 已可导入现有 TCP 代理，但导入后的设备还缺少可重复同步、变更检测、导入记录和端口冲突诊断。
- Wave 8 已交付 SSH/SFTP/VNC 基础服务，Wave 9 先补齐设备级凭据字段和 frps 运维闭环，为后续真实批量 SSH 执行做数据基础。
- 服务器真实运行需要更明确的安全配置、密钥配置、日志暴露和中文错误提示。

## 目标

Wave 9 的目标是把平台从“本地开发可用”和“单点功能可测”推进到“服务器环境可验收、API 可系统测试、frps 导入可运维、设备凭据可配置”。

交付后，管理员应能：

1. 在服务器上快速判断 `502`、后端未启动、代理目标错误、WebSocket 转发失败等部署问题。
2. 使用 Postman Collection 完成登录、Token 自动注入和主要后端 API 回归测试。
3. 对 frps Dashboard 中的现有代理执行预览、导入、重复同步、冲突诊断和结果追踪。
4. 为已导入的 frps 设备配置设备级 SSH 凭据，为后续真实 SSH 连接和批量执行打基础。
5. 在中文前端界面中看到稳定、可理解、不会误触发退出登录的错误提示。

## 范围

### 1. 服务器部署诊断与健康检查

必须新增一个面向部署验收的诊断能力，覆盖以下内容：

- 后端健康状态：`/api/health` 返回服务名、版本、数据库连接状态、运行环境摘要。
- 配置摘要：返回后端当前读取到的关键非敏感配置，例如 API 前缀、数据库类型、远程网关主机、VNC 网关主机、文件后端、SSH/VNC 超时时间。
- 代理链路诊断文档：明确如何分别测试后端直连、前端 Vite 代理、Nginx 代理、WebSocket 代理。
- 服务器排查脚本或命令集合：输出端口监听、进程、健康检查、Nginx 配置片段、最近错误日志。
- 前端在 API 502/503/504 时显示“后端服务不可达或代理配置错误”，不得清空登录态，除非返回状态是 401 或 403。

验收重点：

- 后端未启动时，文档能引导定位到 `127.0.0.1:8000` 不通。
- 前端代理目标端口错误时，文档能引导定位到 `VITE_API_PROXY_TARGET` 或 Nginx `proxy_pass` 错误。
- 远程 SSH/VNC 网关配置错误时，后端返回中文错误，不泄露密码或私钥路径。

### 2. API 测试资产与 Postman 验收闭环

必须把 API 测试资产作为平台交付的一部分，而不是临时手工文件。

需求：

- 在仓库内提交可维护的 Postman Collection 或生成脚本，建议路径为 `docs/postman/edge-platform.postman_collection.json`，不能只依赖下载目录中的临时 JSON。
- 登录接口的 Tests 脚本自动保存 `access_token` 和 `refresh_token`。
- Collection 级别配置 Bearer Token：`{{access_token}}`。
- Collection 变量至少包含：`base_url`、`username`、`password`、`access_token`、`refresh_token`。
- 健康检查、登录、刷新 Token 接口不强制 Bearer Token；其他受保护接口默认继承集合级鉴权。
- 提供“最小回归顺序”：健康检查、登录、当前用户、设备列表、frps 预览、监控概览、日志列表。
- README 或 `docs/api.md` 中增加 Postman 使用说明和常见错误说明，例如把脚本误放到 Pre-request Script 导致 `pm.response` 为空。

验收重点：

- 新导入 Collection 后，先运行登录接口，再运行设备列表接口，无需手工复制 Token。
- Token 过期后可运行刷新 Token 接口更新变量。
- Collection 中的服务器地址可以通过变量改成测试服务器地址。

### 3. frps 导入运维化

现有 frps 导入功能应从“一次性导入”升级为“可重复运维同步”。

需求：

- 保留当前 Dashboard 配置输入：地址、用户名、密码、SSH 端口范围、VNC 端口范围、项目号、部署位置。
- 支持导入前预览以下状态：
  - 新设备：平台尚未存在，可导入。
  - 已存在：按 `device_sn`、SSH 端口或 VNC 端口匹配到已有设备。
  - 端口冲突：端口池被其他设备占用。
  - 缺失 VNC：发现 SSH 代理但未发现对应 VNC 代理。
  - 离线代理：frps Dashboard 返回非 online 状态。
- 支持重复同步时更新已有设备的在线状态、代理名称、标签和缺失 VNC 提示，但不得覆盖管理员手工修改的设备名称、项目号和位置，除非用户显式选择覆盖。
- 每次导入或同步必须写入操作日志，记录发现数量、新增数量、跳过数量、冲突数量和失败原因。
- 前端导入结果应以中文表格展示，不只显示一条成功消息。

验收重点：

- 对同一 frps Dashboard 连续执行两次导入，第二次应显示“已存在/跳过或同步”，不得重复创建设备。
- 当端口池中已有冲突端口时，接口返回冲突明细。
- 当 Dashboard 认证失败或不可达时，前端显示中文错误，并保留已输入配置。

### 4. 设备级凭据字段与远程连接准备

Wave 9 不直接实现真实批量 SSH 命令执行，但必须补齐设备级凭据字段，解决当前只能依赖全局环境变量配置 SSH 凭据的问题。

需求：

- 设备模型新增或扩展 SSH 凭据字段，至少支持：
  - SSH 用户名，默认使用 `ztl`。
  - SSH 密码，本轮默认值为 `123456`，直接保存至数据库；后续 Wave 再考虑加密保存。
  - 私钥路径/私钥引用可作为后续扩展，本轮不强制实现。
  - SSH 凭据类型，例如 `password`、`private_key`。
  - 可选的私钥 passphrase 引用。
- 凭据字段必须支持旧数据库兼容迁移，现有设备启动时不得报错。
- frps 导入已存在设备时，不得覆盖管理员手工配置的凭据。
- frps 新设备导入时，默认 SSH 用户为 `ztl`，默认 SSH 密码为 `123456`。
- 后端 API 响应不得返回明文密码、私钥内容或 passphrase。
- 前端设备创建/编辑表单应支持配置 SSH 用户和凭据类型；敏感输入使用密码框。
- 设备列表或详情只展示凭据是否已配置，不展示敏感值。
- README 和 API 文档必须说明设备级凭据与全局环境变量的优先级：
  - 设备级凭据优先。
  - 未配置设备级凭据时，可回退到服务器环境变量。

验收重点：

- 新建或编辑设备时可以保存 SSH 用户 `ztl` 和凭据配置状态。
- 新建或导入设备时默认 SSH 密码为 `123456`。
- 查询设备详情不返回明文密码。
- 从 frps 导入的新设备默认 SSH 用户为 `ztl`。
- 旧数据库迁移后已有设备可正常加载。

### 5. 后续真实批量 SSH 执行需求冻结

Wave 9 只冻结后续真实执行所需的数据基础，不交付真实批量命令执行。以下内容保留为 Wave 10 或后续需求的输入：

Wave 8 已提供可复用的 SSH 服务，Wave 9 应把批量更新任务从模拟执行推进为可选的真实 SSH 命令执行。

需求：

- 批量任务新增或复用执行模式：
  - `dry_run`: 只筛选设备并生成计划，不连接设备。
  - `ssh_command`: 对目标设备执行管理员配置的命令。
- 真实执行必须逐设备记录：
  - 设备 ID、设备名称、SSH 端口、开始时间、结束时间。
  - 执行状态：pending、running、success、failed、skipped、cancelled。
  - stdout、stderr 的摘要或受限长度内容。
  - 失败原因，例如连接超时、认证失败、命令退出码非 0、设备缺少 SSH 端口。
- 支持失败策略：
  - `continue_on_failure`: 单设备失败不阻断后续设备。
  - `stop_on_failure`: 首个失败后跳过剩余设备。
- WebSocket 任务进度应推送真实设备执行状态。
- 默认命令必须安全，测试可使用只读命令，例如 `hostname`、`whoami`、`uptime`，不得默认执行破坏性命令。
- 命令输入必须有基本限制：非空、长度限制、中文错误提示；危险命令拦截可作为后续增强，但本轮至少不得在文档或示例中推荐破坏性命令。

后续验收重点：

- 对 1 台可访问测试设备执行 `hostname`，任务状态变为 success，并记录输出摘要。
- 对 1 台不可访问设备执行任务，状态变为 failed，失败原因为中文可读。
- 混合成功和失败设备时，`continue_on_failure` 能继续执行后续设备。

### 6. 安全与配置收敛

需求：

- 后端必须支持通过环境变量配置 JWT 密钥、管理员默认密码、远程网关地址、VNC 网关地址、SSH 密码或私钥路径。
- README 必须明确生产环境不得使用默认 `admin/admin` 和 `change-me-in-production`。
- 日志和 API 响应不得输出 Dashboard 密码、SSH 密码、私钥内容或完整 Token。
- frps Dashboard 凭据在前端表单中使用密码输入框，提交失败后不得在错误日志中明文展示。
- 文档建议限制 frps Dashboard 访问来源，避免公网暴露弱口令。

验收重点：

- 错误响应只包含必要原因，不包含敏感值。
- 服务器部署说明覆盖环境变量示例。

### 7. 中文界面与错误提示一致性

需求：

- 新增和改动的前端文本必须使用中文。
- API 错误、表单校验、导入结果、批量任务执行状态必须使用中文展示。
- 502/503/504 不应导致登录状态被清除。
- 401/403 才触发登录态失效处理。
- Postman 文档、README 新增说明使用中文。

验收重点：

- 使用错误密码登录只显示一条中文错误。
- 后端不可达时显示代理/服务不可达提示，不跳回登录页。
- frps 导入失败、批量 SSH 失败均显示中文原因。

## 数据与接口约束

- 应优先复用现有 `devices`、`update_tasks`、`operation_logs`、`port_pool` 模型和服务。
- 如需要新增字段，必须提供 SQLite 旧库兼容迁移，不能让既有服务器数据库启动失败。
- WebSocket 鉴权沿用现有 Token 查询参数模式。
- Postman Collection 应与当前 `/api` 前缀保持一致。
- frps 端口默认规则继续采用 SSH `12001-17000`、VNC `17001-22000`，VNC 端口与 SSH 端口偏移 5000，但 UI/API 仍允许用户覆盖范围。
- frps 导入设备默认 SSH 用户为 `ztl`，默认 SSH 密码为 `123456`。
- frps 同步允许管理员显式选择覆盖已有设备的项目号和部署位置；默认策略仍是不覆盖，只有用户选择覆盖时才更新。
- 服务器正式测试访问路径以 Nginx 单域名反向代理为准；同一域名负责前端静态资源、`/api` 和 `/api/ws` 转发。Vite dev server 仅作为开发调试路径。
