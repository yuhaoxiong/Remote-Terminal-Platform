# Wave 23 需求文档：管理员系统设置与数据库配置覆盖

> 阶段：`requirement_doc`
> 状态：已确认，待按执行计划实现
> 基线：Wave 22 已完成前端企业级运维后台重构与体验统一，平台已具备多用户、角色权限、远程连接、文件管理、批量更新、定时任务、告警通知、系统诊断和操作日志能力。本轮目标是在不破坏现有配置和部署方式的前提下，新增仅管理员可见的系统设置页面，并让部分后端运行参数支持数据库覆盖、分组保存、重启提示和变更审计。

## 1. 背景

当前平台大部分运行配置来自后端 `Settings` 和环境变量，例如远程网关地址、SSH/VNC 超时、文件存储、调度器、默认设备凭据、JWT 生命周期和凭据加密密钥状态。

这种方式适合初始部署，但在服务器测试和后续运维中存在以下问题：

- 部分配置调整需要登录服务器修改环境变量或服务配置，操作成本高。
- 前端只能在诊断页查看有限配置摘要，不能形成配置管理闭环。
- 缺少“数据库配置覆盖 > 系统配置 > 代码默认值”的统一读取规则。
- 修改配置后是否立即生效、是否需要重启、是否已经重启应用，缺少明确状态。
- 敏感配置、配置变更历史和重启操作缺少系统设置级别的审计视图。

Wave 23 的目标是引入“后端注册表白名单 + 数据库覆盖值 + 管理员系统设置页”的配置管理能力。系统设置用于管理平台运行参数，不用于替代部署层的所有环境变量，也不开放任意 key。

## 2. 目标

Wave 23 完成后，系统应达到以下状态：

1. 前端新增“系统设置”一级页面，仅 `admin` 可见。
2. 后端新增系统设置 API，仅 `admin` 可读取、保存、恢复默认和触发重启。
3. 系统设置只允许修改后端注册表白名单内的配置项，不允许管理员新增任意 key。
4. 配置读取顺序统一为：`数据库覆盖值 > 系统配置/环境变量 > 代码默认值`。
5. 后端启动时加载系统配置和数据库覆盖值，合并为内存有效配置；保存后刷新内存缓存。
6. `DATABASE_URL` 和 `CREDENTIAL_ENCRYPTION_KEY` 保持启动级配置，不允许数据库覆盖。
7. `JWT_SECRET_KEY` 第一版不纳入可编辑白名单，只做状态展示和风险提示。
8. 敏感配置使用 `CREDENTIAL_ENCRYPTION_KEY` 加密保存；未配置加密密钥时，禁止保存敏感配置，但允许保存非敏感配置。
9. 系统设置按分组单独保存，保存重启后生效配置时必须弹窗告知用户。
10. 系统设置页新增“重启服务”按钮，支持通过 systemd 托管场景触发后端延迟退出并由 systemd 拉起。
11. 重启操作必须高风险二次确认，并在无法确认 systemd 托管时拒绝执行。
12. 需要重启的配置保存后持久化 `pending_restart` 状态，服务重启并成功加载配置后自动清除。
13. 新增独立系统设置变更历史表，同时继续写入现有操作日志。
14. 诊断页继续允许 `operator` 查看有限只读、脱敏状态，但系统设置页和系统设置 API 完全限制为 `admin`。
15. 数据库结构变更通过 Alembic 迁移推进，不在业务启动时静默创建表。

## 3. 非目标

本轮不实现以下内容：

- 不开放任意自定义配置 key。
- 不支持从前端修改 `.env`、systemd unit 或服务器环境变量。
- 不让 `DATABASE_URL` 通过数据库覆盖。
- 不让 `CREDENTIAL_ENCRYPTION_KEY` 通过数据库覆盖。
- 不实现 JWT 密钥轮换，不在系统设置页编辑 `JWT_SECRET_KEY`。
- 不管理具体 Webhook URL、密钥、Header 或通知通道内容，这些仍归属告警中心。
- 不实现系统设置历史一键回滚。
- 不实现多实例配置同步、分布式缓存刷新或消息总线。
- 不实现完整配置审批流。
- 不把系统设置前端隐藏作为安全边界，真正权限以后端 `403` 为准。

## 4. 配置读取与生效模型

### 4.1 读取顺序

有效配置必须按以下顺序解析：

```text
数据库覆盖值 > 系统配置/环境变量 > 代码默认值
```

要求：

- 数据库只保存覆盖值，不复制所有默认配置。
- 如果某个 key 没有数据库覆盖值，则回退到系统配置或环境变量。
- 如果系统配置也没有提供，则回退到代码默认值。
- 前端必须展示每个配置项当前来源：`数据库覆盖`、`系统配置`、`默认值`。

### 4.2 启动加载与缓存

要求：

- 后端启动时先读取系统配置/环境变量，再读取数据库覆盖值。
- 合并结果写入 `app.state.settings` 或等价的 `EffectiveSettings` 内存对象。
- 业务请求优先读取内存有效配置，不在每次请求中重复查询数据库。
- 管理员保存系统设置后，后端重新计算有效配置并刷新当前进程内存缓存。
- 如果后续部署多进程或多实例，另行增加配置版本轮询或消息通知机制，本轮不做。

### 4.3 非法配置处理

服务启动时如果读取到数据库非法配置，按风险分级处理：

| 类型 | 处理方式 |
| --- | --- |
| 启动关键配置非法 | 拒绝启动，并在日志中明确指出配置项 |
| 普通运行配置非法 | 跳过数据库覆盖值，回退系统配置/默认值，并记录诊断风险 |
| 敏感配置解密失败 | 拒绝使用该敏感配置，回退系统配置；无可用回退时标记对应能力不可用 |

系统设置页应展示“配置异常”状态，并引导管理员恢复默认值或重新保存。

## 5. 后端注册表白名单

后端需要维护系统设置注册表，例如 `SYSTEM_SETTING_REGISTRY`。每个配置项至少包含：

```text
key
name
description
category
value_type
default_value 或 default_source
editable
secret
requires_restart
runtime_effective
validation
options
```

要求：

- 前端从后端接口读取注册表和当前有效值。
- 管理员只能编辑注册表中 `editable=true` 的配置项。
- 后端必须对 key、类型、范围、枚举和敏感字段进行校验。
- 不允许通过 API 保存注册表之外的 key。
- 敏感字段不回显明文，只返回 `configured=true/false`、来源和更新时间等状态。

## 7. 数据库与迁移需求

本轮通过 Alembic 新增数据库结构。

### 7.1 `system_settings`

建议字段：

```text
id
key
category
value_type
value_json
secret_value_encrypted
source
is_secret
requires_restart
pending_restart
is_valid
invalid_reason
updated_by
updated_at
created_at
```

要求：

- `key` 唯一。
- 非敏感值存储结构化 JSON 或等价安全格式。
- 敏感值只存密文，不存明文。
- `pending_restart` 需要持久化。
- 恢复默认值时删除或停用数据库覆盖记录，使有效值回退到系统配置或代码默认值。

### 7.2 `system_setting_changes`

建议字段：

```text
id
setting_key
category
action
old_source
new_source
old_value_snapshot
new_value_snapshot
is_secret
requires_restart
pending_restart_after_change
actor_user_id
actor_username
client_ip
created_at
```

要求：

- 保存、恢复默认、触发重启、启动后应用待重启配置都应记录。
- 敏感字段不记录明文旧值和新值，只记录 `***` 或状态摘要。
- 系统设置页的“变更历史”从该表读取。
- 现有操作日志仍记录系统设置相关操作摘要。

## 8. API 需求

建议新增系统设置 API 分组：

```http
GET /api/system-settings/schema
GET /api/system-settings/effective
PUT /api/system-settings/groups/{group_key}
POST /api/system-settings/{key}/reset
GET /api/system-settings/changes
POST /api/system-settings/restart
```

权限要求：

- 以上接口全部要求 `admin`。
- 未登录返回 `401`。
- 登录但非 `admin` 返回 `403`。
- `operator` 不应通过系统设置 API 枚举配置。

行为要求：

- `schema` 返回注册表、分组、字段类型、校验规则、是否敏感、是否需要重启。
- `effective` 返回当前有效值、来源、是否数据库覆盖、是否待重启；敏感字段不回显明文。
- `PUT /groups/{group_key}` 按分组保存，事务内校验并写入覆盖值。
- 保存需重启配置时，返回 `requires_restart=true` 和待重启项列表。
- `reset` 删除数据库覆盖值，回退到系统配置或默认值。
- `changes` 支持分页、配置 key、分组、操作者、时间范围筛选。
- `restart` 执行重启前校验 systemd 托管状态，无法确认时拒绝执行并返回中文错误。

## 9. 重启服务需求

系统设置页需要提供“重启服务”按钮。

### 9.1 前端交互

要求：

- 仅 `admin` 可见。
- 存在待重启配置时，页面顶部显示醒目提示。
- 修改重启后生效配置并保存前，弹窗提示“该配置保存后需重启服务才会生效”。
- 点击“重启服务”必须进行高风险二次确认。
- 建议要求管理员输入固定确认文本：`确认重启`。
- 确认弹窗说明服务将短暂不可用，远程连接、批量任务进度刷新、告警投递等可能受影响。
- 调用重启接口后，前端进入重启等待状态并轮询健康检查。
- 服务恢复后提示“服务已恢复”，并刷新系统设置状态。

### 9.2 后端行为

要求：

- 仅 `admin` 可调用重启接口。
- 后端检测运行环境是否由 systemd 托管。
- 如果无法确认 systemd 托管，拒绝执行并返回中文错误，不退出进程。
- 确认可重启时写操作日志和系统设置变更历史。
- 返回 `202 Accepted` 后延迟 1-2 秒退出当前进程。
- 由 systemd `Restart=always` 或等价配置自动拉起服务。
- 服务启动成功并应用待重启配置后，自动清除 `pending_restart` 状态并记录变更历史。

## 10. 前端页面需求

### 10.1 导航与权限

要求：

- 左侧导航新增“系统设置”入口。
- 入口仅 `admin` 可见。
- `operator` 不显示系统设置入口。
- 如果非 admin 直接访问或接口返回 `403`，显示中文无权限提示，不退出登录。

### 10.2 页面结构

系统设置页建议分为以下区域：

- 顶部状态栏：配置健康、待重启状态、加密密钥状态、systemd 托管状态、重启服务按钮。
- 分组设置：
  - 基础运行或安全摘要。
  - 远程连接。
  - 默认设备凭据。
  - 文件与存储。
  - 调度器。
  - 告警通知。
  - 安全与认证。
- 变更历史。

### 10.3 分组保存

要求：

- 每个分组独立保存。
- 每个分组显示字段来源：`数据库覆盖`、`系统配置`、`默认值`。
- 来自数据库覆盖的配置支持“恢复默认值 / 删除数据库覆盖值”。
- 保存失败时在当前分组显示中文错误。
- 空值、类型错误、范围越界、枚举非法应在前端和后端双重校验。
- 敏感字段只允许输入新值，不回显旧值。
- 未配置 `CREDENTIAL_ENCRYPTION_KEY` 时，敏感字段禁用保存并显示风险提示。

### 10.4 状态与反馈

要求：

- 加载中使用 Skeleton 或 Loading。
- 无配置项时显示中文 Empty。
- 保存成功显示中文提示。
- 保存需重启项后显示待重启 Alert。
- 恢复默认值需要确认。
- 重启服务需要高风险二次确认。
- 所有错误提示中文。

## 11. 诊断页联动

诊断页继续允许 `admin` 和 `operator` 查看有限只读状态，但必须脱敏。

建议新增或完善摘要：

- 系统设置表是否存在。
- 数据库覆盖项数量。
- 待重启配置数量。
- 是否配置凭据加密密钥。
- 是否检测到 systemd 托管。
- 是否存在非法数据库覆盖值。
- JWT 是否使用默认密钥。

要求：

- 不展示密码、Token、私钥、Webhook 明文密钥或完整数据库连接串。
- `operator` 只能看到安全摘要，不能看到系统设置详情。

## 14. 验收标准

Wave 23 通过验收需满足：

1. 只有 `admin` 可见并访问系统设置页。
2. 系统设置 API 只允许 `admin` 使用，`operator` 返回 `403`。
3. 配置读取顺序符合 `数据库覆盖值 > 系统配置/环境变量 > 代码默认值`。
4. 数据库只保存白名单配置覆盖值，未知 key 被拒绝。
5. 分组保存、恢复默认值、变更历史和操作日志均可用。
6. 敏感配置加密保存，未配置加密密钥时禁止保存敏感配置。
7. API、前端和变更历史均不泄露敏感明文。
8. 需重启配置保存后持久化待重启状态。
9. 服务重启成功并重新加载配置后自动清除待重启状态。
10. 重启服务按钮具备 systemd 检测、高风险确认、`202` 返回和健康轮询。
11. 诊断页保留 operator 可见的有限脱敏摘要。
12. Alembic 迁移、后端测试、前端测试、类型检查和构建通过。
13. README、API/部署/手工测试文档按实际实现更新。
14. `git diff --check` 通过。

## 15. 已确认决策

1. 读取顺序为 `数据库覆盖值 > 系统配置/环境变量 > 代码默认值`。
2. 尽量减少非必须环境变量。
3. `DATABASE_URL` 和 `CREDENTIAL_ENCRYPTION_KEY` 保持启动级配置，不允许数据库覆盖。
4. 启动时加载到内存，保存后刷新内存缓存。
5. 系统设置只允许修改后端注册表白名单内的配置项。
6. `JWT_SECRET_KEY` 第一版不纳入可编辑白名单，只做状态展示和风险提示。
7. 默认设备 SSH 凭据允许编辑，敏感值加密存库。
8. 未配置 `CREDENTIAL_ENCRYPTION_KEY` 时，不允许保存敏感配置，但允许保存非敏感配置。
9. Webhook 只配置通用运行参数，不管理具体 URL、密钥或通道。
10. 按分组单独保存。
11. 新增独立变更历史表，同时写操作日志。
12. 系统设置页和系统设置接口只允许 `admin`，诊断页继续允许 `operator` 查看有限只读状态。
13. 数据库迁移按 Alembic 推进。
14. 需重启配置保存前弹窗告知，保存后持久化待重启状态。
15. 系统设置页增加“重启服务”按钮。
16. 重启服务由 systemd 托管后端进程，后端返回 `202` 后延迟退出。
17. 检测不到 systemd 托管时，后端拒绝重启并返回中文错误。
18. 点击重启服务需要高风险二次确认。
19. 服务重启成功后自动清除待重启状态。
20. 启动读取到非法数据库设置时，按配置风险分级处理。
21. Wave23 白名单范围按本文冻结，先不继续扩大。
